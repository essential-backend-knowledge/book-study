# Ch.3 성능을 좌우하는 DB 설계와 쿼리

> 주니어 백엔드 개발자가 반드시 알아야 할 실무 지식 — 3장 발표 자료
>
> 핵심 메시지: **DB 성능은 쿼리 튜닝이 아니라 설계 단계에서 결정된다. "어떤 API가 어떤 데이터를 반복 조회하는가"를 먼저 보고 설계해야 한다.**

---

## 1. 성능에 핵심인 DB

DB 성능 문제는 연동하는 **모든 서버 성능에 영향**을 준다. DB 성능 문제로 전체 서비스가 먹통이 되는 상황은 흔히 발생한다. 하지만 DB 자체가 문제인 경우는 많지 않다. 오히려 **DB를 잘못 사용해서 발생한 문제**가 더 많다.

### 풀 스캔(Full Scan)이란

WHERE 절 조건에 맞는 인덱스가 없을 때 테이블 전체를 순차적으로 읽는 것이다. 데이터가 적을 때는 문제없어 보이지만, **데이터 증가 후 기하급수적으로 느려진다.** DB CPU 사용률이 치솟는 주범이다.

**실제 사례 — H 서비스**
```
코로나 재택근무 → 가입자 급증 → 출근 8:40 출근태그 시간에 DB CPU 90% 초과
→ 10초 타임아웃 에러 폭발

원인: 호출 빈도가 높은 기능에 인덱스 없는 풀스캔 쿼리 존재
→ 처음엔 데이터가 적어서 빨랐지만, 쌓이면서 폭발
```

## 2. 조회 트래픽을 고려한 인덱스 설계

### 원칙 1. 조회 패턴 기반 설계

인덱스 설계의 출발점은 **"어떤 API가 어떤 조건(WHERE)으로 조회하는가"**이다.

```sql
-- 이 쿼리가 자주 실행된다면 → category 칼럼에 인덱스 필요
SELECT id, category, writerId, title
FROM article
WHERE category = 10
ORDER BY id DESC
LIMIT 20;

-- writerId로도 자주 조회한다면 → writerId 인덱스도 추가
SELECT id, category, writerId, title
FROM article
WHERE writerId = 10
ORDER BY id DESC;
```

**판단 순서**:
1. API가 어떤 조건(WHERE)으로 조회하는가
2. 해당 조건 칼럼에 인덱스가 있는가
3. 데이터 규모가 늘어도 버틸 수 있는가

---

### 원칙 2. 선택도(Selectivity) 고려

- **선택도 높음** = 고유값이 많음 = 인덱스 효율 좋음
- **선택도 낮음** = 고유값이 적음 = 일반적으로 인덱스 효율 나쁨

```
gender 칼럼 (M, F, N 3가지 값)
→ 전체 100만 건 중 M이 50만 건이면, 인덱스 써도 50만 건 탐색 → 비효율
```

**예외 — 선택도가 낮아도 유용한 경우**:
```
status 칼럼 (W대기, P처리중, C완료) — 작업 큐 테이블
→ 대부분 C(완료), 극소수만 W(대기)
→ WHERE status = 'W' 쿼리가 반복 실행된다면 인덱스 필수
```

---

### 원칙 3. 단일 vs 복합 인덱스 판단

사용자당 데이터 규모로 판단한다:

```
사용자당 수천 건 이하 → 단일 인덱스(userId)로 충분
사용자당 수만 건 이상 → 복합 인덱스(userId, activityDate) 필요

예:
- 주 1회 방문, 활동 5건/일 → 5년 후 1,500건 → 단일 인덱스 OK
- 매일 방문, 활동 30건/일 → 1년 후 10,000건 이상 → 복합 인덱스 필요
```

---

### 원칙 4. 커버링 인덱스 활용

쿼리 결과를 인덱스 내에서 완성할 수 있으면 실제 테이블 I/O가 불필요하다. 인덱스에 필요한 칼럼이 모두 포함되어 있으면 데이터를 읽는 과정이 생략되어 쿼리 실행 시간이 빨라진다.

```sql
-- activityDate만 인덱스 → 인덱스로 행 선택 후 테이블에서 데이터 조회 필요
-- (activityDate, activityType) 복합 인덱스 → 인덱스 안에서 바로 집계 가능
SELECT activityDate, activityType, COUNT(activityType)
FROM activityLog
WHERE activityDate = '2024-07-28'
GROUP BY activityType;
```

---

### 인덱스는 필요한 만큼만

효과가 적은 인덱스를 추가하면 **오히려 성능이 나빠질 수 있다.** 인덱스는 조회 속도를 빠르게 하지만 INSERT, UPDATE, DELETE 시에는 인덱스 관리 비용이 추가된다.

- 새 인덱스를 추가하기 전에 **기존 인덱스로 커버 가능한지** 먼저 검토
- 요구사항을 약간 변경하면 기존 인덱스를 활용할 수 있는 경우도 많다
- 같은 칼럼에 대한 중복 인덱스를 만들지 않도록 주의

---

## 3. 조회 성능 개선 방법

### 미리 집계하기

count, sum 같은 집계를 **조회 시점에 매번 실행**하면 성능 문제가 발생한다.

```
설문 목록 조회 시:
- 목록 조회 1번
- 답변자 수 서브쿼리 30번 (각 10만 건 카운트)
- 좋아요 수 서브쿼리 30번 (각 1만 건 카운트)
→ 총 61번 쿼리, 약 4.51초 소요
```

**해결**: 집계 데이터를 미리 계산해서 별도 칼럼에 저장한다.

```sql
-- 답변 추가 시 카운트 증가
INSERT INTO answer VALUES (...);
UPDATE survey SET answerCnt = answerCnt + 1 WHERE surveyId = ?;

-- 목록 조회 — 서브쿼리 없이 직접 칼럼 참조
SELECT s.id, s.subject, answerCnt, likeCnt
FROM survey s ORDER BY id DESC LIMIT 30;
```

> 약간의 정합성 불일치를 감수하더라도, 실시간 집계용 칼럼을 추가하는 것이 실용적이다.

---

### Cursor 기반 목록 조회 (ID 기준)

OFFSET 방식은 **지정한 오프셋만큼 데이터를 세는 시간**이 추가된다. 데이터가 많아질수록 뒤쪽 페이지로 갈수록 느려진다.

```sql
-- OFFSET 방식 — 99,990개를 세고 나서 10개 조회
SELECT ... FROM article ORDER BY id DESC LIMIT 10 OFFSET 99990;

-- ID 기준 방식 — 인덱스로 id=9984를 바로 찾아 10개 조회
SELECT ... FROM article WHERE id < 9985 ORDER BY id DESC LIMIT 10;
```

모바일 화면의 무한 스크롤 방식에는 ID 기준 조회가 더 적합하다.

---

### 조회 범위를 시간 기준으로 제한

대부분의 기능은 **최신 데이터를 주로 조회**한다. 3년 전 공지사항은 아무도 읽지 않는다.

```sql
-- 월별 주문 내역 조회
SELECT ... FROM order
WHERE customerId = 100
  AND orderDt >= '2024-08-01' AND orderDt < '2024-09-01'
ORDER BY orderDt DESC;
```

최신 데이터 위주로 조회하게 변경하면 **DB 메모리 캐시 효율도 향상**된다. 최신 데이터가 캐시에 적재될 확률이 높기 때문이다.

---

### 전체 개수 세지 않기

데이터가 많아질수록 `COUNT(*)` 실행 시간이 증가한다. 조건에 해당하는 모든 데이터를 탐색해야 하기 때문이다.

조회 속도가 느려지고 있는데 그 원인이 count 쿼리에 있다면, **전체 개수를 화면에 표시하지 않는 방향으로** 서비스 담당자와 협의하는 것을 고려해야 한다.

---

### 오래된 데이터 삭제 및 분리 보관

데이터 개수가 일정하게 유지되면 쿼리 실행 시간도 일정 수준으로 유지된다.

- 서비스 DB에는 180일 이내 데이터만 유지
- 181일 이전 데이터는 별도 저장소로 분리 보관
- 사용자 서비스는 적절한 성능 유지, 관리 시스템은 필요한 데이터 조회 가능

## 4. 실무 쿼리 함정

### 함정 1. 타입 불일치 조인

```sql
-- user.userId = INT, push.receiverId = VARCHAR
-- DB가 각 행마다 타입 변환 수행 → receiverId 인덱스 무효화

-- 해결: 비교 대상 칼럼의 타입을 맞춰서 비교
SELECT u.userId, u.name, p.*
FROM user u, push p
WHERE u.userId = 145
  AND CAST(u.userId AS CHAR CHARACTER SET utf8mb4)
      COLLATE 'utf8mb4_unicode_ci' = p.receiverId
  AND p.receiverType = 'MEMBER';
```

문자열 타입이라면 **캐릭터셋(charset)** 도 같은지 확인해야 한다.

---

### 함정 2. 운영 환경 DDL 주의

MySQL은 테이블을 변경할 때 **새 테이블 생성 → 원본 데이터 복사 → 원본 교체** 방식을 사용한다. 복사 중에는 DML이 블로킹된다.

**실제 사례**:
```
열거 타입 칼럼 변경 → Online DDL 미사용 → 수천만 건 복사 시작
→ 10분이 지나도 안 끝남 → 관련 테이블 모두 DML 블로킹
→ 점검 모드 전환 → 몇 시간 후 완료 → 복제 DB도 추가 대기 필요
→ 단 한 줄의 DDL이 8시간 서비스 중단
```

**예방법**:
- 운영 DB DDL 전 반드시 DBA 또는 팀 협의
- 가능하면 Online DDL (`ALGORITHM=INPLACE`) 사용
- 데이터 규모가 크면 점검 시간 확보 후 진행

---

### 함정 3. 대량 처리 쿼리 분할

집계 쿼리가 3시간째 실행 중인 상황: `group by + 집계함수 + 서브쿼리 + 대량 데이터 + INSERT...SELECT`

**해결 패턴**:
1. `INSERT...SELECT`를 `SELECT + INSERT`로 분리
2. 시간 구간별로 나눠서 처리 (하루씩 실행 후 합산)
3. 하루 단위도 길면 1시간, 10분 단위로 더 세분화

---

### 함정 4. DB 최대 연결 개수

```
API 서버 4대 x 커넥션 풀 30개 = 필요 커넥션 120개
DB 최대 연결 개수 = 100개 → 20개 연결 실패 발생
```

DB CPU 사용률 > 70%인 상태에서 연결 수를 늘리면 오히려 악화된다. 먼저 캐시 구성, 쿼리 튜닝으로 부하를 낮춘 뒤 증가시켜야 한다.

---

## 5. 트랜잭션 설계

### 실수 1. 트랜잭션 없는 다중 변경

```java
// 트랜잭션 없음
memberDao.insert(member);        // 성공
contractDao.insert(contract);    // 성공
userStatusDao.update(userStatus); // 실패!
// → contract에는 데이터 추가됨, userStatus는 미반영 → 데이터 불일치
```

> 여러 회사에서 트랜잭션 없는 다중 변경 코드를 실제로 자주 본다. "당연히 걸겠지"라고 생각하기 때문에 코드 리뷰에서도 놓치는 경우가 많다.

---

### 실수 2. 예외 전파로 인한 잘못된 롤백

```java
// 메일 발송 실패 → 회원 가입 전체 롤백
@Transactional
public void join(JoinRequest join) {
    memberDao.insert(member);
    mailClient.sendMail(...); // RuntimeException 발생 시 전체 롤백
}
```

메일 발송에 실패했다는 이유로 회원 가입 자체가 실패하는 것은 의도한 동작이 아니다.

```java
// 부가 기능 예외는 분리 처리
@Transactional
public void join(JoinRequest join) {
    memberDao.insert(member);
    try {
        mailClient.sendMail(...);
    } catch (Exception ex) {
        log.warn("메일 발송 실패, 회원 가입은 정상 처리", ex);
    }
}
```

**원칙**: 외부 I/O(메일, API 호출)는 트랜잭션 범위 밖 또는 예외 분리. 핵심 기능 실패 ≠ 부가 기능 실패.

## 6. DB 확장과 캐시 서버

### Primary-Replica 구조

조회 트래픽 비중이 높은 서비스에서, 변경은 주 DB, 조회는 복제 DB를 사용해 처리량을 증가시킨다.

**주의 — 상태 변경 기능에서 복제 DB 조회 금지**:
- 주 DB → 복제 DB 간 데이터 복제에는 **지연(Replication Lag)**이 발생한다
- 주 DB에서 데이터를 변경한 직후 복제 DB에서 조회하면 **변경 전 데이터가 반환**될 수 있다
- INSERT, UPDATE, DELETE 쿼리를 실행하는 기능에서 변경 대상 데이터를 조회해야 한다면 **주 DB에서 SELECT**해야 한다

## 내 실무 경험

### 소상공인 프로젝트 — DB 설계 단계에서 조회 패턴을 먼저 본 경험

POS, 카드 매출, 홈택스, 배달앱 등 이기종 채널 거래 데이터를 수집·조회하는 도메인에서, 데이터가 쌓일수록 조회 성능이 직결되었다.

**3계층 분리 설계**:
```
t_tnx_record          ← 채널별 원천 데이터 그대로 적재 (raw)
      ↓
t_tnx_deduped         ← 중복 제거 + 소스 우선순위 적용 후 최종 조회 기준
      ↓
t_tnx_summary_daily   ← 일/주/월 단위 사전 집계 (실시간 계산 부담 제거)
t_tnx_summary_monthly
```

- 대량 데이터 구간에 날짜 기준 RANGE 파티션 적용
- API는 원천 데이터를 매번 조합하지 않고 가공된 읽기 기준 테이블 직접 조회

### 소상공인 프로젝트 — 빠른 조회 ≠ 올바른 조회

같은 거래가 POS + 카드 + 홈택스 + 배달앱에서 중복 유입되는 문제. 단순 INSERT만 하면 매출 합계가 부풀려진다.

**채널별 중복 판단 기준 + 소스 우선순위 정책**을 수립하고, `t_tnx_deduped`에 `is_visible`, `is_include` 필드로 일관된 기준을 적용했다.

> 실무의 DB 설계는 빠르게 읽는 것만이 아니라, **어떤 데이터를 진실(source of truth)로 간주할 것인가를 정하는 일**이다.

## 챕터 핵심 내용 요약

1. **풀 스캔은 시한폭탄이다.** 데이터가 적을 때는 문제없지만, 쌓이면 어느 순간 기하급수적으로 느려진다.
2. **인덱스는 조회 패턴 기반으로 설계한다.** 어떤 API가 어떤 조건으로 조회하는지가 출발점이다.
3. **단일 vs 복합 인덱스 판단 기준은 사용자당 데이터 규모다.** 수천 건 이하면 단일, 수만 건 이상이면 복합 인덱스가 필요하다.
4. **집계는 미리 계산하라.** 조회 시점에 count/sum을 매번 실행하면 트래픽이 몰릴 때 급격히 느려진다.
5. **OFFSET 대신 커서 기반 조회.** 모바일 무한 스크롤에 적합하고, 뒤쪽 페이지에서도 성능이 일정하다.
6. **운영 DB DDL은 재앙의 씨앗이다.** 수천만 건 테이블 변경은 반드시 DBA와 협의하고 점검 시간을 확보해야 한다.
7. **트랜잭션 경계를 명확히 하라.** 외부 I/O와 DB 작업을 분리 설계하고, 핵심 기능 실패와 부가 기능 실패를 구분해야 한다.
8. **복제 DB에서 변경 대상 데이터를 조회하지 마라.** Replication Lag으로 인한 데이터 불일치가 발생한다.

---

## 본인이 인상 깊었던 부분 + 이유

**"빠른 조회 ≠ 올바른 조회"**라는 관점이 가장 인상 깊었다.

이기종 채널 거래 데이터를 다루면서, 인덱스를 아무리 잘 설계해도 **같은 거래가 중복 노출되면 매출 합계가 부풀려지는 문제**를 경험했다. 조회가 빠른 것과 조회 결과가 올바른 것은 완전히 다른 문제였다. 채널별 중복 판단 기준과 소스 우선순위 정책을 먼저 수립하고, 그 위에 조회 성능 최적화를 얹어야 했다.

책에서도 집계 칼럼의 비정규화를 다루면서 "약간의 불일치를 감수하더라도 실용적 선택을 하라"고 말한다. 성능과 정합성의 트레이드오프를 의식적으로 판단하는 것, 그리고 **어떤 데이터를 source of truth로 볼 것인가**를 정하는 것이 DB 설계의 본질이라는 점을 다시 한번 확인했다.

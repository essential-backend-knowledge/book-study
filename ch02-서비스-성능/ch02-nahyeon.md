# Ch.2 느려진 서비스, 어디부터 봐야 할까

> 주니어 백엔드 개발자가 반드시 알아야 할 실무 지식 — 2장 발표 자료
>
> 핵심 메시지: **성능 문제의 95%는 코드 로직이 아니라 I/O(DB, 외부 API)에서 발생한다.**

---

## 1. 성능이란 무엇인가

### 응답 시간 (Response Time)

사용자의 요청을 처리하는 데 걸리는 시간이다. 성능이 나쁘다는 것은 결국 **결과가 늦게 표시되는 것**으로 체감된다.

- **TTFB (Time to First Byte)**: 응답 데이터 중 첫 번째 바이트가 도착할 때까지 걸린 시간
- **TTLB (Time to Last Byte)**: 응답 데이터의 마지막 바이트가 도착할 때까지 걸린 시간

응답 데이터 크기가 작으면 TTFB와 TTLB 차이가 크지 않지만, 파일 다운로드처럼 전송 데이터가 크거나 네트워크가 느리면 차이가 커진다.

**응답 시간이 사업에 미치는 영향**

| 지연 | 영향 |
|------|------|
| 100ms 증가 | Amazon 매출 1% 감소 |
| 400ms 지연 | Google 검색 횟수 0.6% 감소 (복구 후에도 감소세 지속) |

> 응답 시간이 길어지면 트래픽과 매출이 줄어든다. 줄인다고 반드시 증가하진 않지만, 길어지면 확실히 나빠진다.

---

### 처리량 (Throughput / TPS)

**TPS(Transaction Per Second)** — 초당 트랜잭션 수. 시스템이 단위 시간당 처리하는 작업량이다.

- 최대 TPS를 초과하는 요청이 들어오면 → 대기 발생 → 응답 시간 증가
- TPS를 높이는 2가지 방법:
    1. 동시에 처리할 수 있는 수를 늘린다 (서버 증설, 스레드 풀 확장)
    2. 요청 하나당 처리 시간을 줄인다 (쿼리 최적화, 캐시 적용)

> 성능 개선의 출발점: **현재 TPS와 응답 시간을 먼저 측정**하고, 목표를 설정한 뒤 개선안을 도출해야 한다.

---

### 실제 서버 처리 시간 분포

전체 처리 시간 **348ms** 기준 실측 사례:

| 구간 | 시간 | 비율 |
|------|------|------|
| 외부 API 연동 | 186ms | **53%** |
| DB 연동 (SQL 6회) | 101ms | **29%** |
| 내부 API 연동 | 44ms | 13% |
| 서비스 로직 수행 | 17ms | **5%** |

API + DB 연동이 전체의 **95%**를 차지하고, 서비스 로직(if, for 등)은 **5%에 불과**하다. API 연동이 없는 경우에도 DB 연동이 전체 실행 시간의 70~80% 이상을 차지한다.

> **핵심**: 성능 개선 = 로직 최적화가 아니라 **I/O 최적화**

## 2. 병목은 어디서 오는가

성능 문제는 서비스 초기에는 잘 발생하지 않는다. 사용자가 늘고, 트래픽이 증가하고, 데이터가 쌓이면서 점차 나타난다. 방치하면 어느 순간 전체 시스템이 멈추는 심각한 문제로 터진다.

### 병목 진단 순서

#### STEP 1. DB 커넥션 풀
풀 고갈 → 커넥션 대기 → 응답 시간 급증. DB CPU 사용률이 낮은데 느리면 커넥션 풀 고갈을 의심해야 한다.

#### STEP 2. 슬로우 쿼리
```sql
-- MySQL slow query log 활성화
SET GLOBAL slow_query_log = 'ON';
SET GLOBAL long_query_time = 1; -- 1초 이상 걸리는 쿼리 기록
```
잡힌 쿼리는 `EXPLAIN`으로 실행 계획 분석 → 인덱스 누락 여부 확인

#### STEP 3. 캐시 Hit Rate
Redis Hit Rate **< 90%** → 캐시 전략 재검토. TTL, 무효화 정책이 적절한지 점검.

#### STEP 4. GC 로그 (JVM)
- **Stop-The-World** 빈도와 시간 확인
- GC가 잦으면 응답 시간이 불규칙하게 튄다
- 힙 덤프 분석으로 대량 객체 생성, 메모리 누수 탐지
- Discord가 Go → Rust로 전환한 이유도 GC 문제

#### STEP 5. 외부 API 응답 시간
타임아웃 설정이 적절한지, 서킷 브레이커 패턴 적용 여부 확인

---

### 수직 확장과 수평 확장

**수직 확장(Scale-Up)**: CPU, 메모리, 디스크 등의 자원을 증가시키는 것. 급한 불을 끄는 방법이다. 클라우드 환경에서 비교적 빠르게 시도할 수 있지만, 트래픽이 지속 증가하면 한계가 있다.

**수평 확장(Scale-Out)**: 서버를 추가로 투입해 TPS를 높이는 방법. 로드 밸런서가 트래픽을 분산한다.

> **주의**: 실제 병목 지점이 어디인지 파악하는 게 중요하다. DB에 문제가 있는데 서버를 추가하면 **불에 기름을 붓는 격**이다. DB에 가해지는 부하가 더 커지고 성능 문제는 더 악화된다.

## 3. DB 커넥션 풀

서버와 DB는 네트워크 통신으로 연결된다. 매 요청마다 연결을 생성하고 종료하면 그 시간이 전체 응답 시간에 영향을 준다. DB 커넥션 풀은 **이미 연결된 커넥션을 재사용**해서 이 문제를 해결한다.

### 핵심 설정 4가지

| 설정 | 설명 | 실무 포인트 |
|------|------|------------|
| **커넥션 풀 크기** | 미리 생성할 커넥션 수 | DB CPU 사용률 80% 이상이면 늘리지 말고, 오히려 유지하거나 줄여야 한다 |
| **커넥션 대기 시간** | 풀에 커넥션이 없을 때 기다리는 최대 시간 | HikariCP 기본 30초는 너무 길다. 0.5~3초 이내 권장 |
| **최대 유휴 시간** | 사용되지 않는 커넥션을 유지할 시간 | DB의 비활성화 유지 시간보다 짧게 설정 |
| **최대 유지 시간** | 커넥션의 전체 생존 시간 | 무한대로 설정하지 말 것 |

### 커넥션 대기 시간이 길면 생기는 문제

대기 시간이 30초라면 → 사용자가 기다리다 취소 후 재요청 → 동시 처리 요청 수 급증 → 서버 부하 악화. **대기 시간을 1초로 짧게 설정하면** 빠르게 에러를 응답하고 서버 부하를 일정 수준으로 유지할 수 있다.

## 4. 서버 캐시

DB 서버를 확장하지 않고도 응답 시간과 처리량을 개선하는 방법이다. 캐시에 데이터를 저장해두면 동일한 요청 시 DB 대신 캐시에서 데이터를 읽어 응답한다.

### 적중률(Hit Rate)과 삭제 규칙

- **적중률** = 캐시에 존재한 건수 / 캐시에서 조회를 시도한 건수
- 적중률이 높을수록 DB 연동이 줄어들고, 응답 시간 감소 + 처리량 증가 + DB 부하 감소
- 캐시가 가득 찼을 때 삭제 규칙: **LRU** (가장 오래전 사용), **LFU** (가장 적게 사용), **FIFO** (먼저 추가된 것)

### 로컬 캐시 vs 리모트 캐시

| 구분 | 로컬 캐시 | 리모트 캐시 (Redis 등) |
|------|----------|----------------------|
| 속도 | 매우 빠름 (메모리 직접) | 빠름 (네트워크 1회) |
| 일관성 | 다중 서버 시 불일치 위험 | 모든 서버 동일 |
| 적합 데이터 | 변경 적은 설정값, 코드 | 가격, 상태, 세션 등 |
| 재시작 시 | 캐시 데이터 소실 | 데이터 유지 |

**선택 기준**:
- **민감한 데이터** (가격, 게시글 내용) → 변경 즉시 캐시 무효화 + 리모트 캐시
- **비민감한 데이터** (인기글 목록) → TTL 기반 주기적 갱신, 로컬 캐시 사용 가능

### 캐시 사전 적재 (Cache Warming)

트래픽이 순간적으로 급증하는 패턴이라면, 캐시에 데이터를 미리 넣어두는 것을 고려해야 한다. 푸시 알림 직후 150만 명이 동시 접속하는 상황에서 캐시 적중률이 0%에 가까워지면 DB 부하가 폭발한다.

## 5. GC와 대용량 데이터 처리

### GC(Garbage Collector)의 성능 영향

메모리 사용량 증가 → GC 실행 빈도 증가 → Stop-The-World → 응답 시간 증가 → TPS 감소

### 대용량 데이터 처리 원칙

```java
// 위험 — 30MB 파일 x 100명 동시 접근 = 3GB 메모리 필요
byte[] bytes = Files.readAllBytes(Path.of("path"));
out.write(bytes);

// 안전 — 100명 동시 접근 시 800KB만 필요 (3,750배 차이)
InputStream is = Files.newInputStream(Path.of("path"));
byte[] buffer = new byte[8192];
int read;
while ((read = is.read(buffer, 0, 8192)) >= 0) {
    out.write(buffer, 0, read);
}
```

- 조회 범위 제한 (기간, 건수 MAX 설정 필수)
- 대량 데이터 = 스트림 또는 페이징 처리
- `Files.readAllBytes()` 운영 환경에서 사용 금지

## 6. 응답 데이터 압축과 CDN

### 응답 데이터 압축 (gzip)

텍스트 데이터(HTML, CSS, JS, JSON)를 gzip으로 압축하면 **70% 이상 크기 감소**. 이미 압축된 파일(JPEG, ZIP)은 재압축 효과 없음. 클라우드 환경에서는 트래픽 = 비용이므로 비용 절감 효과도 있다.

### CDN (Content Delivery Network)

지역별 에지 서버에 콘텐츠를 캐시해서 제공한다.
- 오리진 서버 트래픽 감소
- 지리적으로 가까운 서버에서 제공 → 빠른 응답
- 트래픽 비용 절감

---

## 7. 대기 처리 (Waiting System)

순간적으로 트래픽이 폭증할 때 (콘서트 예매 등):

| 방법 | 장점 | 단점 |
|------|------|------|
| 서버, DB 증설 | 직접적 처리량 증가 | 짧은 피크를 위한 고정 비용 증가, DB는 쉽게 줄일 수 없음 |
| 대기 처리 시스템 | 서버 증설 없이 안정적 운영, 새로고침 시 순번 밀림 → 불필요한 요청 자체를 줄임 | 사용자 대기 필요 |

> 대규모 트래픽을 처리하기 위해 지불해야 할 비용은 결코 적지 않다. 비용 관점에서 대기 제어는 매우 합리적인 방식이다.

## 내 실무 경험

### 법령 번역 서비스 — 병목을 코드가 아닌 구간으로 본 경험

파일 업로드 → AI 연동 → 상태 반영 → 파일 등록이 한 요청에 묶인 구조에서 응답 지연이 발생했다. 처음에는 "서비스 로직이 복잡한가?"부터 코드를 봤지만, 구간별로 분석해보니:

- 외부 AI 연동 응답 지연 → 압도적 1위
- DB 상태 반영 → 그 다음
- 서비스 로직 자체 → 거의 무시 가능 수준

> 성능 문제를 볼 때 코드보다 I/O 구간을 먼저 확인하는 습관이 생겼다. 느리다 = 서비스 로직이 복잡하다 → 이 공식은 틀렸다.

## 챕터 핵심 내용 요약

1. **성능 지표는 응답 시간과 처리량(TPS)이다.** 응답 시간은 주로 DB와 외부 API 연동 시간이 지배한다.
2. **병목 진단 순서**: DB 커넥션 풀 → 슬로우 쿼리 → 캐시 Hit Rate → GC 로그 → 외부 API 응답 시간. 추측하지 말고 측정하라.
3. **수직 확장은 급한 불 끄기, 수평 확장은 병목 지점 파악 후에.** DB가 병목인데 서버를 늘리면 상황이 악화된다.
4. **DB 커넥션 풀 설정이 성능을 좌우한다.** 커넥션 대기 시간을 짧게 설정해야 서버 부하 폭증을 막을 수 있다.
5. **캐시는 전략적으로 사용한다.** 민감도에 따라 로컬/리모트를 구분하고, 트래픽 패턴에 따라 사전 적재를 고려한다.
6. **대용량 데이터는 스트림 처리가 필수.** `readAllBytes()`는 운영에서 금지. 조회 범위 제한은 기본이다.

## 본인이 인상 깊었던 부분 + 이유

**"서비스 로직 수행 시간은 전체의 5%에 불과하다"**는 실측 데이터가 가장 인상 깊었다.

법령 번역 서비스에서 성능 문제가 발생했을 때 처음 본 것이 서비스 로직이었다. "코드가 복잡해서 느린 거 아닐까?"라는 직관이 먼저 작동했기 때문이다. 하지만 구간별 시간을 측정해보니 외부 AI 연동이 압도적 1위였고, 서비스 로직은 무시할 수 있는 수준이었다.

이 경험이 책의 "API + DB 연동이 95%"라는 수치와 정확히 겹친다. 성능 문제를 만나면 **코드 리팩토링부터 떠올리는 습관**이 있었는데, 실제로는 **I/O 구간을 먼저 측정하는 것**이 훨씬 효과적이라는 걸 직접 체감했다. "추측하지 말고 측정하라"는 원칙이 왜 중요한지 몸으로 배운 셈이다.
